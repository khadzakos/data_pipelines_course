# Управление секретами в Kubernetes
Автор: Николай Хадзакос

Для работы сравниваются нативные **Secrets**, **HashiCorp Vault**, **External Secrets Operator**, плюс коротко **Sealed Secrets**.

---

## Критерии

| Критерий | Пояснение |
|----------|------------|
| Где хранится | etcd, отдельный сервис, файлы в поде |
| Безопасность | шифрование, кто может прочитать |
| Аудит | можно ли понять, кто запрашивал секреты |
| Ротация | обновление паролей/ключей без ручной возни |
| GitOps | удобно ли класть манифесты в Git |
| Сложность | сколько сил на установку и поддержку |

---

## По инструментам (по факту)

### Нативные Kubernetes Secrets

Встроенный объект: кладёшь данные в `Secret`, под забирает через env или volume.  
**Плюсы:** ничего не ставишь сверх кластера.  
**Минусы:** лежит в etcd — нужно шифрование at-rest, иначе бэкапы etcd опасны; ротация сама не делается; в Git plaintext класть нельзя.  
**Когда:** учебные/небольшие проекты или база, если etcd уже шифруют и RBAC настроен.

### HashiCorp Vault

Отдельный сервис для секретов: хранение, политики, динамические секреты, аудит. В K8s подключают через agent injector, CSI и т.д.  
**Плюсы:** сильный аудит, TTL, динамические пароли к БД и т.п.  
**Минусы:** тяжело админить (HA, бэкапы), долго вникать, по времени/лицензиям дороже.  
**Когда:** много команд, жёсткие требования по безопасности и учёту доступа.

### External Secrets Operator (ESO)

Оператор тянет секреты из AWS/GCP/Azure/Vault и создаёт обычные `Secret` в кластере.  
**Плюсы:** одна схема для разных облаков; приложения как раньше читают `Secret`.  
**Минусы:** типично секрет всё равно оказывается в etcd; зависимость от внешнего API и прав (IAM и т.д.).  
**Когда:** секреты уже живут в облачном Secret Manager, нужно просто подтянуть их в K8s.

### Sealed Secrets

В Git кладут зашифрованный манифест; в кластере контроллер делает из него обычный `Secret`.  
**Плюсы:** удобно для Argo CD / Flux, секрет не светится в репозитории открытым текстом.  
**Минусы:** важен ключ контроллера; динамики как у Vault нет.  
**Когда:** GitOps и не хочется отдельный секрет-сервис в облаке.

## Сравнительная таблица 1 — по критериям

Условно: «+» лучше / проще, «−» хуже / сложнее.

| | Где хранится | Безопасность (если настроить) | Аудит | Ротация | GitOps | Сложность внедрения |
|--|--------------|-----------------------------|-------|---------|--------|-------------------|
| Native Secret | etcd | средне (нужно KMS/шифрование etcd) | слабый | руками | не из Git | низкая |
| Vault | свой сервис | сильная | сильная | сильная | так себе | высокая |
| ESO | etcd + внешний источник | зависит от облака и etcd | у внешнего хранилища | зависит от провайдера | норм | средняя |
| Sealed Secrets | etcd после расшифровки | зависит от ключа | слабый | руками/процесс | очень удобно | низкая |

## Сравнительная таблица 2 — когда что брать

| Инструмент | Коротко про сценарий |
|------------|----------------------|
| Native Secret | Минимум магии, но не забыть про шифрование etcd |
| Vault | Нужен центр секретов, аудит, TTL, сложная организация |
| ESO | Секреты уже в облаке, в K8s хотим обычные `Secret` |
| Sealed Secrets | Всё в Git, GitOps |
---

## Вывод

Одного «лучшего» нет: **нативные Secret** — база, но без шифрования etcd слабовато. **Vault** — мощно, но тяжело по поддержке. **ESO** — удобный мост из облака. **Sealed Secrets** — про GitOps.
